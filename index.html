128K Prototype

The proof of concept for 128K was created in Tumult Hype, but it quickly became clear that the application would not be sophisticated enough to handle all the functionality needed for the project. Nevertheless, my questions and bug reports quickly caught the attention of Hype’s developers, and I’ve been a part of their beta program ever since, helping improve the app.

The full site is being developed with AngularJS, Sass, Grunt (with a number of plugins), Karma, and Protractor. To facilitate this, a reusable build/test system — called ng-foundation — is currently being developed, and should see its first commit to GitHub very soon. (The source for the site itself is in a private repository on BitBucket, however the "system files" will detail much of how the site works.)

The site is intended to emulate much of the functionality of Macintosh System 1, as released in 1984, using nothing but modern web technologies. It will serve as my primary online presence for years to come (eventually replacing this portfolio page) as well as a playground where I can experiment with developing different types of web apps and advance the boundaries of my knowledge.

A Warm Gun

A Warm Gun was a blog I created and ran from 2005–2009. Originally intended as my personal learning platform for tableless CSS-based layouts—still a fairly new thing in late 2004, when work on the site began—this design was actually the second (and final) that the site used. The first design used flat HTML files, but this design ran on WordPress 2 as a custom theme, which also provided me with an opportunity to learn a little bit of PHP so I could customize the WordPress install and a few plugins to fit my particular needs.

The site was a moderate success, and the design proved to be fairly long-lived, but the advent of responsive web design and a growing repertoire of mobile devices hitting the scene eventually rendered it outdated. Since my passion lied more with web development than with blogging, I decided to start a new project to learn RWD and shuttered the site in 2010.

  • Client: None (Personal)
  • Date: October 2006
  • Services: Web Development, Graphic Design

Enemy Planet Main Menu (iPad)

The main menu graphics were created in Sketch and then sliced into PNGs and imported into Xcode. Rather than having a complex nested menu, I worked hard to come up with a design that would reduce the number of menu views to an absolute minimum. For example, rather than having a separate view for the four main settings (number of players, SFX on/off, BGM on/off, and in-game help on/off), these were bubbled up to the front of the menu. Since the main menu could quickly be accessed from anywhere in the game, players would have easy access to those four main settings at any time.

An earlier version of the design had text labels for the four buttons on the left (play, downloads, credits/indicia, and achievements), but after much deliberation I decided it would be better to remove the text labels and make the main menu entirely icon-based. Doing so cut down the amount of localization that would need to be done for the app, and made the buttons more Fittsian by making the target areas larger. (Not to mention it looked better.)

  • Client: Anvil Games
  • Date: May 2012
  • Service: UI Design

Enemy Planet Wall Placement UI (iPhone)

Gameplay for Enemy Planet was split into three phases: wall placement, missile control tower placement, and battle. During wall placement, players were given Tetris-style pieces that they would use to surround (and thus claim) territory. Claimed territory could then be used in the next phase of play for place missile control towers. This placement UI, minus the ability to rotate, was also used to place those towers.

To rotate a piece, players simply place their thumb and forefinger on the screen and twist. Moving a piece just involved tapping and dragging anywhere on the screen—the game would just translate the path your finger takes to the piece directly, illuminating the appropriate direction arrows that follow the piece around the screen at the same time. If any portion of a piece was placed over a tile that could not be built on, the portion of the piece over that tile would turn from green to red.

The place button would also follow the piece around the screen. While normally oriented above the wall piece, the place button would automatically rotate to a free side if it hit a screen edge. Tapping the timer would pause the game and bring up a menu view that would overlay the gameplay layer. The “nuke” button would only be active during the battle phase.

  • Client: Anvil Games
  • Date: July 2010
  • Services: UI Design, Graphic Design

Enemy Planet Dialog UI (iPhone)

During level selection, players were given two buttons: one for level-specific settings, and another to access the achievements available for that level. Icons for each achievement would appear as only silhouettes until the achievement was earned. Tapping an achievement icon would display that achievement’s info in the main part of the dialog, while tapping the Game Center icon would open an in-game Game Center view so that users could see all the achievements for the game at once, as well as leaderboards.

  • Client: Anvil Games
  • Date: May 2010
  • Services: UI Design, Graphic Design

Enemy Planet concept sketch: outposts

Players in Enemy Planet are in control of a set of outposts which they must defend by shooting at attacking aliens (or rival colonies in multiplayer). There are two types of outposts, which we refer to as the “Mushroom” and the “Thimble”, based on their designs. The Mushroom is the player’s “home” outpost—that is, the initial outpost the player selects in a stage.

The outposts occupy 2x2 tiles, but have some visual “spillover” into adjacent tiles. A spinning radar antenna sits atop both types of outpost, as does a red light sprite that blinks on and off. Outposts have two states: claimed and unclaimed. Claimed enables the blinking light and spinning radar animations, while unclaimed has no light and the radar stuck in one position. Claimed outposts also have an invisible forcefield around them which becomes visible when the outpost is hit. This forcefield protects the outpost from damage. The colors match the colors of the wall forcefields—blue for Player 1 and red for Player 2. While the forcefield is activated in a claimed outpost, unclaimed outposts are not as lucky and can suffer some amount of damage before they are destroyed.

  • Client: Anvil Games
  • Date: March 2010
  • Service: Concept Sketches

Star Fleet Battles SSD: Federation CA

In 2008, I introduced a friend of mine to the classic wargame Star Fleet Battles. Based on the classic Star Trek series from the 1960s, the game uses schematics called SSDs for each ship. Each box on the SSD represents a specific piece of equipment, which has certain requirements and capabilities. I was unsatisfied with the design SSDs that come with the game, so I decided to create my own for use in our games.

This is the Federation Heavy Cruiser, which is based on Star Trek’s U.S.S. Enterprise. As you can see from the original SSD below, the new one is a significant improvement, while being functionally identical. Since the ship we used in our game session was the basic, non-upgraded model, those additional systems were not included on my SSD.
